# Phase 4.5 · Investigate

Scaffold 결과물 (`willa-preview` 5175) 의 품질을 **Serve 전에 조사** · 사용자 필수 확인.

## 왜 필요한가

Scaffold 완료 후 willa 가 "품질 좋다" 착각하는 경우 많음. 실제로 보면:
- 아이콘 placeholder 남아있음
- 상태 4종 중 hover/active 누락
- 간격/위계가 plan 과 불일치
- 모션 토큰 정의만 있고 적용 0
- 접근성 기초 누락

이걸 **사용자에게 보여주기 전에 (Serve)** 잡아야 함.

## 조사 방법 (3개 중 2개 이상 조합)

### 1. Visual QA · 스크린샷 + plan 대조 ⭐

```
1. http://localhost:5175/ 캡처 (브라우저 수동 · 스크린샷)
2. plan-latest.md 의 토큰/레이아웃 표 읽기
3. 실제 렌더 vs plan 비교:
   - 팔레트 hex 일치?
   - 컴포넌트 높이/너비 토큰 일치?
   - 상태별 시각 차이 구분되는가?
   - 폰트 위계 3단계 구분되는가?
4. 어긋난 항목 리스트업
```

### 2. Self-check · 코드 품질 체크리스트 ⭐

```
✓ 아이콘 lucide-react 사용 (이모지 0개)
✓ shadcn/ui 컴포넌트 최소 5종 사용 (Button/Badge/ScrollArea/Tooltip/...)
✓ Tailwind 클래스 사용 · inline style 최소화
✓ 모든 interactive 요소: default/hover/active/disabled 구현
✓ 모션 토큰 실제 적용 (transition · duration · easing)
✓ 4px grid spacing 준수
✓ 접근성: tooltip · aria-label · focus ring
✓ TypeScript 에러 0
✓ 상태 완전성: 로딩/빈/에러 상태 존재 (references/states-empty-error.md 기준)
```

체크리스트 결과 표 작성 · 각 항목 ✅/❌/⚠ 표시.

### 2b. NNG Heuristic 감사 (선택 · 심화)

`references/nng-heuristics.md` 의 10 heuristic 중 이번 plan 에 관련있는 3-5개 선택 감사:

- 1 Visibility of system status → 세그 진행률 · 로딩 표시 있나
- 3 User control and freedom → Undo · Cancel · Back 가능한가
- 5 Error prevention → 신경 충돌 경고 · 삭제 확인
- 6 Recognition rather than recall → 아이콘 단독 쓰지 말고 라벨 병기
- 7 Flexibility and efficiency → 단축키 + 시각 진입점 병행
- 8 Aesthetic and minimalist design → 불필요 요소 제거

기준은 NNG 원전 · 적용은 Will3D 맥락으로 해석.

### 3. frontend-design 서브에이전트 감사 (선택)

```
Skill("frontend-design:frontend-design",
  args: "이번 plan 방향성: {Operating Theatre / Layered Workspace / ...}.
         5175 렌더 검토. 개선 제안 5개 이내. 특정 디자인 시스템 강제 금지. ~200 words.")
```

감사 결과를 `.willa/audit-pre-serve-{ts}.md` 에 저장.

## 사용자 필수 확인 (AskUserQuestion)

조사 결과를 바탕으로 사용자에게 **무조건** 질문:

```
header: "조사 결과"
질문: "조사 결과 · 어떻게 진행할까?"
옵션:
  1. OK · Serve 로 진행
     설명: 품질 충분 · 보고 메시지 생성 후 종료
  2. 개선 필요 · Scaffold 로 되돌려서 고치기
     설명: 조사에서 발견한 이슈 N개 수정 후 재조사
  3. 부분 수정 · 특정 항목만
     설명: (사용자가 텍스트로 지정) · 해당 부분만 수정 후 Serve
  4. 조사 결과만 보고 결정 보류
     설명: 조사 결과 문서만 남기고 현재 상태로 Serve · 사용자 나중에 판단
```

**스킵 금지**. AskUserQuestion 없이 Serve 진입 = 품질 게이트 실패.

`.willa/investigate-{YYYYMMDD-HHmm}.md` 형식:

```markdown

## 3. frontend-design 감사 (선택)
```

## 분기

- **OK** → Phase 5 Serve
- **개선 필요** → Phase 4 Scaffold 로 복귀 (수정 대상 명시) → 다시 Phase 4.5
- **부분 수정** → 수정 후 Phase 5 로
- **보류** → 현재 상태 Serve · investigate 문서만 보관

## 루프 제한

- 조사 ↔ Scaffold 반복 최대 3회
- 3회 후에도 개선 필요 → Serve 로 강제 진행 + 미해결 항목 명시

## 주의

- 조사는 **사용자 시간을 아껴주는** 게이트. 조사 없이 Serve 하면 사용자가 직접 발견 → 반복
- 조사 자체가 너무 길어지지 않게 · Visual QA + Self-check 2개만으로도 충분한 경우 많음
